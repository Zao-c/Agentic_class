# ADR-0006：报警码必须通过结构化设备范围门控

- 状态：已采纳
- 日期：2026-07-14

## 决策

故障诊断不再把通用 RAG 的相似文本命中视为报警码证据。Agent 必须先用规范化报警码、品牌、型号和可选控制器版本查询结构化表，再根据匹配状态决定继续或转交：

```text
必要槽位完整
→ lookup_error_code
→ manual_retrieval 仅作补充证据
→ 结构化匹配与冲突门控
→ 安全约束
→ 保存诊断状态
→ 输出安全核对项或人工转交
```

`exact_match` 和经来源核验的 `brand_match_model_unverified` 可以生成辅助说明；后者必须显式提示型号适用性未验证。`ambiguous`、`model_mismatch`、`brand_mismatch`、`not_found` 和草稿记录不得生成确定性诊断。

## 原因

- 报警编号可能在不同品牌、型号或控制器版本中具有不同含义。
- 通用向量相似度不能证明编号和设备范围准确匹配。
- 故障建议涉及设备与人身安全，需要把来源、风险和禁做事项作为结构化字段，而不是依赖生成文本临时判断。

## 当前范围

首批只录入课程资料中可追溯的 ABB `10036` 与 `38213`。资料没有标出具体型号，因此两条记录的 `equipment_models` 为 `*`，状态只能达到品牌匹配、型号未核验。该限制在回答、Trace 和已知限制中公开保留。

## 验收证据

`diagnosis_eval_v1.json` 覆盖信息不足、正常诊断、资料缺失、安全绕过、高风险报警、多轮补全和品牌冲突。运行 `scripts/evaluate_diagnosis.py` 生成原始 JSON/Markdown 报告；同码冲突和型号不匹配另由 pytest 固定覆盖。
